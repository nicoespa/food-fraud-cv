# food-fraud-cv

[![Open In Colab](https://colab.research.google.com/assets/colab-badge.svg)](https://colab.research.google.com/github/nicoespa/food-fraud-cv/blob/main/notebooks/food_fraud_cv_colab.ipynb)

Claim-conditioned refund-evidence fraud detection for food-delivery photos.
Final project — Ciencia de Datos (UdeSA). Replaces `meal-dispatch-uncertainty`.

> **Cómo se corre:** el slice liviano y los tests, local (`uv`). El **pipeline real**
> (datos + difusión + fine-tune) se corre en **GPU de Colab** — abrí el badge de arriba o
> `notebooks/food_fraud_cv_colab.ipynb`. Ver `docs/03-real-pipeline.md`.

**Problem.** Customers edit delivery photos with AI (add mold, make food look
raw/spoiled) to trigger fraudulent refunds. The right question is **not** "is this
image AI-generated?" but **claim-conditioned verification**: does this photo prove
the damage the customer claims?

**Thesis.**
1. A generic AI-image detector **fails** at refund verification — false positives on
   *real* damage, misses *fake* damage, and transfers poorly across generators. (We
   replicate FraudBench's finding with public data.)
2. **Best accuracy ≠ best business decision.** The model with the best accuracy/AUC
   is **not** the one that minimizes expected refund cost (FN = paid fraud, FP =
   rejected honest customer). A **calibrated** probability feeding a **cost-sensitive**
   policy (approve / deny / human-review) wins. This is the 7-vs-10 insight.

Three classes, layered system:
- **Layer 1 — Data** (`src/data`, `src/generation`): build a 3-class corpus.
  `genuine-undamaged` + `genuine-damaged` from public food datasets; **we generate**
  `fake-damaged` ourselves via diffusion/inpainting, across several generators with a
  held-out generator for cross-generator generalization.
- **Layer 2 — Detection** (`src/detection`): models producing a **calibrated fraud
  probability** conditioned on the claim (zero-shot baseline → fine-tuned CNN/ViT →
  + forensic signals → fusion; MLLM reasoning is stretch).
- **Layer 3 — Decision** (`src/decision`): **cost-sensitive** policy over the score —
  auto-approve / auto-deny / route to **human review**. Connects to Rappi's
  refunds-backoffice (maker-checker).

> Honesty note (for the report): `fake-damaged` is **synthetic, generated by us** (as
> FraudBench did — they didn't release their dataset either). The generators define
> the domain, so we measure **cross-generator** generalization (held-out generator)
> and declare the limitation. 100% public data — **no Rappi data**.

## Run from zero
Requires [`uv`](https://docs.astral.sh/uv/). Python is pinned to 3.12.

```bash
# 1. light env: runnable slice (classical generators + forensic features + sklearn)
uv sync --extra ml

# 2. STAR EXPERIMENT end-to-end (builds synthetic corpus, trains, evaluates):
uv run --extra ml python scripts/run_experiment.py --config configs/default.yaml --rebuild

# tests
uv run --extra ml --extra dev pytest -q

# scale paths (heavy, GPU): fine-tune backbones / diffusion generation
uv sync --extra dl    # torch/torchvision/timm  → src/detection/finetune.py
uv sync --extra gen   # diffusers/transformers   → src/generation/diffusion.py
```

## Status

13 tests verdes. El experimento estrella demuestra empíricamente las dos tesis:
- **Tesis 1:** el detector AIGC genérico (zero-shot) es inútil (PR-AUC ~0.47) e inunda
  de falsos positivos sobre daño real → falla en verificación de reembolso.
- **Tesis 2:** la política cost-sensitive con revisión humana (D2) baja el costo/caso de
  ~0.88 (umbral ingenuo) a ~0.38; mejora significativa (IC95 bootstrap excluye el 0).
